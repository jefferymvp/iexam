# iExam 架构设计说明书

## 1. 文档元信息

- 系统名称：iExam（Web MockExamApp）
- 代码仓库根目录：`c:\develop\iexam`
- 架构风格：Next.js App Router + Supabase BaaS + Client/Server 混合渲染
- 文档目标：将当前实现固化为可审查、可追溯、可演进的工程规范
- 读者对象：产品、后端、前端、测试、运维与安全审计人员

## 2. 背景与范围

### 2.1 业务目标

iExam 面向多组织场景，提供在线练习、正式考试、错题回炉、后台组织与题库运营能力。

### 2.2 本文范围

- 用户认证与会话治理
- 多组织与题库授权模型
- 考试与错题核心业务流程
- 管理后台运营链路（组织、用户、题库、Excel 导入）
- 非功能性要求与已识别风险

### 2.3 非范围

- 移动端原生壳能力
- 离线考试能力
- 第三方支付与计费能力

## 3. 架构总览

### 3.1 分层与职责

1. 表现层（Next.js App Router）
   - 服务端页面负责鉴权与首屏数据装配
   - 客户端组件负责交互、状态与局部数据写入
2. 应用层（Server Actions + Page 逻辑）
   - 登录、注册、登出等认证动作
   - 业务流程编排（组织检查、题库授权检查、题目抽取）
3. 数据与鉴权层（Supabase）
   - Auth：登录态与用户身份
   - PostgreSQL：组织、题库、题目、错题、用户档案
   - RLS：行级数据隔离与权限控制
4. 会话中间层（Next Middleware）
   - 请求级会话刷新
   - 未登录路径重定向

### 3.2 关键技术选型

- Next.js 16 + React 19 + TypeScript
- @supabase/ssr + @supabase/supabase-js
- Tailwind CSS 4
- xlsx（SheetJS）用于 Excel 题库导入

## 4. 组件与模块设计

### 4.1 入口与布局模块

- 根布局：`app/layout.tsx`
  - 注入全局样式与导航
  - 所有页面共享容器与主题布局
- 顶部导航：`components/Navbar.tsx` + `components/NavLinks.tsx`
  - 服务端读取用户与角色
  - 客户端根据路径高亮导航并按管理员身份显示后台入口

### 4.2 认证与会话模块

- 登录/注册动作：`app/login/actions.ts`
  - `login(formData)`：邮箱密码登录
  - `signup(formData)`：邮箱密码注册
  - `logout()`：会话退出
- 会话中间件：`middleware.ts` + `utils/supabase/middleware.ts`
  - 在请求早期刷新会话
  - 拦截未登录访问并重定向 `/login`
- Supabase 客户端工厂
  - 浏览器端：`utils/supabase/client.ts`
  - 服务端：`utils/supabase/server.ts`

### 4.3 首页配置与开考模块

- 服务端首页：`app/page.tsx`
  - 检查登录态
  - 拉取用户已加入组织，传入客户端组件
- 客户端配置面板：`app/HomeClient.tsx`
  - 组织切换
  - 可用题库加载（仅 active 且已授权）
  - 题型、模式、题量配置
  - 生成 query 参数并跳转考试页
- 加入组织：`components/JoinOrg.tsx`
  - 拉取允许加入且用户未加入的组织
  - 插入 `organization_members`

### 4.4 考试引擎模块

- 服务端考试入口：`app/exam/page.tsx`
  - 校验用户是否属于组织
  - 校验组织是否授权题库且题库启用
  - 按题型过滤并随机打乱，截取指定题量
- 客户端考试引擎：`app/exam/ExamEngine.tsx`
  - 支持单选、多选、判断
  - 练习模式即时判题
  - 考试模式交卷统一算分
  - 错题写入 `user_mistakes`（upsert）

### 4.5 错题本模块

- 服务端错题页：`app/mistakes/page.tsx`
  - 联表读取用户错题与题目详情
- 客户端错题引擎：`app/mistakes/MistakeEngine.tsx`
  - 重做错题并即时判定
  - 回答正确后删除对应错题记录

### 4.6 后台管理模块

- 管理入口：`app/admin/page.tsx`
  - 服务端检查 `profiles.role === 'admin'`
- 选项卡容器：`components/admin/AdminTabs.tsx`
  - 组织管理、用户管理、题库管理
- 组织管理：`components/admin/OrgManager.tsx`
  - 组织增删改、是否允许加入
  - 拉起组织配置弹窗
- 组织配置弹窗：`components/admin/OrgConfigModal.tsx`
  - 成员授权（organization_members）
  - 题库授权（organization_banks）
- 用户管理：`components/admin/UserManager.tsx`
  - 读取 `profiles` 并切换角色
- 题库管理：`components/admin/BankManager.tsx`
  - 题库启停、删除
  - Excel 解析导入题库与题目

## 5. 核心业务流程

### 5.1 登录流程

1. 用户在登录页提交表单
2. Server Action 调用 `supabase.auth.signInWithPassword`
3. 登录成功后 `revalidatePath('/', 'layout')` 并重定向首页
4. Middleware 维持后续请求会话有效

### 5.2 开始考试流程

1. 用户在首页选择组织、题库、题型、模式、题量
2. 客户端构造 query 跳转 `/exam`
3. 服务端验证组织成员关系和题库授权关系
4. 服务端拉取题目、随机打散并切片
5. 客户端渲染考试引擎并执行答题流程

### 5.3 错题沉淀与消除流程

1. 判题错误时写入 `user_mistakes`（同用户同题 upsert）
2. 进入错题页读取错题集合
3. 重练正确时删除该错题记录
4. 错题集合实时收敛，形成闭环

### 5.4 后台导入题库流程

1. 管理员上传 Excel
2. 前端用 `xlsx` 转为 JSON
3. 创建题库记录
4. 将行数据转换为 questions 批量插入
5. 导入完成后刷新题库列表

## 6. 数据模型与契约（推断）

以下为基于当前代码推断的核心实体，不等价于最终数据库 DDL。

### 6.1 核心实体

- profiles
  - `id`（用户 ID）
  - `email`
  - `username`
  - `role`（`admin` | `user`）
  - `created_at`
- organizations
  - `id`
  - `name`
  - `description`
  - `allow_join`
  - `created_at`
- organization_members
  - `organization_id`
  - `user_id`
- question_banks
  - `id`
  - `name`
  - `description`
  - `is_active`
  - `created_at`
- organization_banks
  - `organization_id`
  - `bank_id`
- questions
  - `id`
  - `bank_id`
  - `title`
  - `type`（`single` | `multiple` | `judge`）
  - `options`（JSON）
  - `answer`（字符串或数组）
  - `parse`
- user_mistakes
  - `id`
  - `user_id`
  - `question_id`
  - `bank_id`

### 6.2 关键约束

- 用户访问考试前必须满足：
  - 存在 organization_members 关系
  - 存在 organization_banks 关系且题库 `is_active=true`
- 错题记录建议唯一键：
  - `(user_id, question_id)`
- 角色访问约束：
  - `profiles.role='admin'` 才能进入 `/admin`

## 7. 接口契约

### 7.1 页面路由契约

- `GET /`
  - 前置：用户已登录
  - 输出：组织列表与考试配置页
- `GET /exam?orgId&bankId&type&mode&count`
  - 入参：
    - `orgId`：组织 ID（必填）
    - `bankId`：题库 ID（必填）
    - `type`：`all|single|multiple|judge`（建议）
    - `mode`：`show|hide`
    - `count`：题量整数
  - 规则：
    - 参数非法或无权限时回退/提示
  - 输出：考试引擎页面
- `GET /mistakes`
  - 前置：用户已登录
  - 输出：错题列表或空状态
- `GET /admin`
  - 前置：用户已登录且管理员
  - 输出：后台控制台

### 7.2 Server Actions 契约

- `login(formData)`
  - 输入：`email`, `password`
  - 成功：重定向 `/`
  - 失败：返回 `{ error: string }`
- `signup(formData)`
  - 输入：`email`, `password`
  - 成功：重定向 `/`
  - 失败：返回 `{ error: string }`
- `logout()`
  - 成功：重定向 `/login`

## 8. 安全与权限设计

### 8.1 已实现控制

- 请求级会话刷新与未登录重定向
- 后台入口服务端角色检查
- 组织成员与题库授权双重校验
- 客户端只使用 anon key，不暴露 service role key

### 8.2 安全边界

- 最终访问控制依赖 Supabase RLS 策略
- 前端逻辑只负责体验层限制，不构成安全边界

## 9. 非功能设计

### 9.1 可用性

- 页面含加载态、空态和错误提示
- 导入流程有失败告警

### 9.2 性能

- 首屏页面使用服务端拉数
- 考试抽题为“全量取数 + 本地洗牌 + 切片”，题库过大时需要优化

### 9.3 可维护性

- Supabase 客户端封装统一
- 模块边界清晰（考试、错题、后台）
- 仍存在 `any` 类型较多的问题，类型资产不足

## 10. 已识别风险与改进建议

### 10.1 高优先级

1. 题型值不一致风险
   - 首页题型枚举使用 `multi`，但题目类型使用 `multiple`
   - 会导致按题型筛选时多选题查询异常
   - 建议统一为 `multiple`
2. 管理侧直接切换用户角色风险
   - 当前“用户管理”按钮将 `admin` 与 `user`互切，缺少二次确认和审计
   - 建议改为状态封禁/解封与细粒度 RBAC
3. 导入事务一致性风险
   - 题库创建成功但题目插入失败时会留下空题库
   - 建议使用数据库函数或事务式 RPC 保证原子性

### 10.2 中优先级

1. 参数边界缺失
   - `count` 缺少上限和正整数校验
2. 大题库性能压力
   - 目前随机化在应用层完成，建议下沉数据库随机抽样策略
3. 代码基线噪音
   - 目录中存在 `src/app` 模板残留，建议清理避免认知负担

### 10.3 格式转换专项结论

1. Excel 多选答案解析已支持空格与中英文逗号清洗
2. 判断题答案已支持更丰富的“正确”表达映射
3. 仍建议将“格式转换规则”抽离为纯函数并补齐单元测试，避免回归

## 11. 验收标准（AC）
